# 3. Product Definition

Cursivis is a cursor-first productivity layer that works over the user’s current application instead of forcing them to open a separate chatbot.

The core interaction should be:

1. The user selects something or points at something.
2. The user presses a global trigger or activates the orb.
3. Cursivis instantly captures the relevant context.
4. Cursivis determines or asks what the user wants.
5. Cursivis returns a useful result.
6. When appropriate, Cursivis can apply that result through a safe, structured action.

The user should not need to:

* copy content manually;
* switch to ChatGPT;
* paste context;
* explain which app they are using;
* copy the output back;
* repeatedly describe the same screen or form.

The experience should feel like AI has been added directly to the cursor and active workflow.

# 4. Primary Product Modes

The product must contain five distinct primary capabilities:

1. **Context Trigger**

   * selected text analysis;
   * image-region capture when no text is selected;
   * pixel color detection when the user clicks rather than drags.

2. **Realtime Live Mode**

   * low-latency spoken conversation;
   * selection-aware;
   * optionally screen-aware;
   * capable of entering Navigation Guidance Mode;
   * capable of invoking safe tools.

3. **Take Action**

   * structured browser and text actions;
   * manually available from results;
   * directly available through a dedicated global trigger;
   * callable through spoken Live Mode commands.

4. **Smart Dictation**

   * dedicated global trigger;
   * partial transcription;
   * filler removal and self-correction handling;
   * polished text inserted into the active application.

5. **Custom Quick Task**

   * dedicated global trigger;
   * Prompt Optimizer as the default task;
   * customizable into one persistent user-defined task;
   * operates on the current text or supported image context;
   * converts a rough task description into a precise, reviewable finalized instruction before saving;
   * remains subject to the normal Take Action and safety policies.

All five capabilities must also be accessible from the orb.

The product story should remain clear:

* Smart Mode decides the most useful operation dynamically.
* Guided Mode lets the user choose an operation.
* Custom Quick Task runs the user’s personally configured one-key workflow.
* Take Action applies results through validated, deterministic execution.
* Live Mode provides realtime conversation, guidance, and voice-led action requests.

# 6. Context Trigger Behavior

The Context Trigger is the most important interaction and must feel nearly instant.

## When text is selected

Pressing `Ctrl+Alt+O` should:

1. immediately show an orb response state;
2. identify the foreground application;
3. capture the selected text;
4. restore the user’s clipboard if a clipboard fallback was required;
5. route the selection into Smart Mode or Guided Mode;
6. show the result without taking focus away from the source application unnecessarily.

## When no text is selected

Pressing the same trigger should immediately open a region-selection overlay.

The user should drag over:

* an image;
* part of a webpage;
* a chart;
* a UI error;
* a table;
* a product;
* a document;
* any visual region.

The captured region should be analyzed through an OpenAI vision-capable GPT-5.6 model.

## When the user clicks instead of dragging

Treat a click or extremely small selection region as a color-detection operation.

Return:

* hexadecimal color;
* RGB value;
* approximate human-readable color name;
* optional copy button;
* a small color preview.

Do not send a model request for basic pixel color detection.

## Cancellation

Escape or the global Cancel shortcut must close the selection overlay instantly and leave no screenshot or stale temporary file.

# 7. Fast Selection Detection

The current interaction has noticeable selection-detection latency. Treat this as a major product requirement.

Do not rely on long fixed sleeps.

Implement a layered selection pipeline:

## Fast path 1: Chromium extension

When the active application is a supported Chromium browser and the extension is connected:

* request the active page’s current selected text;
* include nearby DOM context only when necessary;
* return immediately;
* do not capture the whole webpage for ordinary text operations.

## Fast path 2: Windows UI Automation

For compatible applications:

* inspect the focused accessibility element;
* use appropriate selection or text patterns;
* retrieve the selected range without changing the clipboard;
* apply strict timeouts and cancellation.

## Fallback: protected clipboard copy

When other paths fail:

1. record the current clipboard sequence number;
2. preserve supported clipboard formats;
3. issue a copy command to the foreground application;
4. wait for an actual clipboard-changed signal rather than sleeping for an arbitrary period;
5. inspect the copied text;
6. restore the previous clipboard;
7. avoid overwriting a clipboard change made independently by the user during the operation.

Use short adaptive timeouts. Never block the UI thread.

## Performance targets

Instrument and report:

* trigger-to-visible-orb time;
* trigger-to-selection-detected time;
* trigger-to-snip-overlay time;
* trigger-to-result-request time;
* end-to-end result time.

Target:

* visible orb feedback: under 50 ms;
* browser fast-path selection: preferably under 100 ms;
* typical selection classification: under 200 ms;
* no-selection transition to snip overlay: preferably under 250 ms;
* no unexplained half-second or one-second fixed pauses.

Create repeatable latency benchmarks and report p50 and p95 values.

# 8. Smart Mode

Smart Mode automatically chooses the most useful operation for the current context.

Examples:

* foreign-language text → translate;
* rough email → polish;
* received email → draft reply;
* code → explain or debug;
* error message → diagnose;
* long prose → summarize;
* question → answer;
* table → extract or explain;
* image → describe or identify;
* UI screenshot → explain the visible issue;
* form → understand the requested information and suggest an action;
* selected prompt → improve the prompt.

Do not implement Smart Mode as an uncontrolled natural-language response.

Use a typed structured result similar to:

```json
{
  "contextType": "email|code|question|prose|image|form|ui|table|other",
  "intent": "translate|reply|rewrite|summarize|explain|debug|extract|identify|answer|other",
  "confidence": 0.0,
  "result": "user-visible result",
  "suggestedAction": {
    "type": "none|replace_selection|insert_text|fill_form|copy|browser_plan",
    "summary": "brief explanation"
  },
  "risk": "none|low|medium|high",
  "requiresConfirmation": true
}
```

Use official structured-output support or strict schema validation.

Rules:

* Do not execute merely because the model generated an action.
* If confidence is below a carefully chosen threshold, fall back to Guided Mode.
* Avoid a separate model request solely for classification when deterministic rules and the main response request can perform the task together.
* Keep prompts concise and task-specific.
* Return useful final content, not descriptions of what the model intends to do.
* Do not include irrelevant metadata, fake confidence language, internal reasoning, or hackathon commentary in user-facing output.

# 10. Custom Quick Task

Add a fifth primary capability named **Custom Quick Task**.

Its default shortcut is:

```text
Ctrl+Alt+Y
```

This feature gives the user one personally configurable operation that can be invoked instantly from anywhere.

The default Quick Task must be **Prompt Optimizer**.

Its default finalized instruction should be equivalent to:

> Rewrite the selected rough prompt into a precise, structured, implementation-ready prompt while preserving every original requirement, constraint, named entity, link, and requested output. Remove unnecessary repetition and ambiguity where safely possible, but do not invent missing details or silently discard requirements.

## Default Prompt Optimizer behavior

When the user selects rough instructions, notes, requirements, or an unstructured prompt and presses `Ctrl+Alt+Y`, Cursivis should:

1. capture the current selection through the normal fast selection pipeline;
2. recognize that the active Quick Task is Prompt Optimizer;
3. preserve the original meaning, requirements, constraints, named entities, links, deadlines, and requested output;
4. remove repetition, unnecessary filler, and avoidable ambiguity;
5. organize the prompt into a clear implementation-ready structure;
6. improve ordering, formatting, specificity, and readability;
7. avoid inventing requirements, credentials, repositories, deadlines, integrations, or product decisions;
8. preserve unusual requirements unless they are clearly accidental duplication;
9. identify genuine contradictions rather than resolving them silently;
10. return only the optimized prompt in the result panel.

Do not add commentary such as:

* “Here is your improved prompt.”
* “Certainly.”
* “I have optimized your request.”
* explanations of the transformation unless the user explicitly asks for them.

The result panel should expose:

* Copy;
* Replace Selection;
* Insert at Cursor;
* Refine;
* More Options;
* Take Action when there is an appropriate and safe execution destination.

If two requirements genuinely conflict, preserve both and mark the conflict clearly in the optimized output instead of choosing one silently.

## User-defined Quick Task

In Settings, allow the user to replace Prompt Optimizer with one persistent custom task.

Examples include:

* convert selected text into concise professional email language;
* translate selected content into English while preserving formatting;
* turn meeting notes into action items;

* rewrite selected text in the user’s concise writing style;

* turn selected text into a LinkedIn post;
* create test cases from selected requirements;
* analyze a selected error and suggest the most likely fix;
* extract structured information from an image;
* summarize a selected research passage in a defined format.

Only one primary saved Quick Task is required for this milestone.

Design the domain and storage interfaces so multiple named Quick Tasks could be supported later without rewriting the core system.

## Quick Task Settings

Settings should include:

* task display name;
* rough task-description input;
* finalized instruction preview;
* supported context types;
* expected output mode;
* whether it may propose a Take Action plan;
* custom hotkey recorder;
* Test Task button;
* Restore Prompt Optimizer Default;

* Cancel.

The user must be able to inspect and manually edit the finalized instruction before saving.

The user must also be able to inspect and edit it later.

## Task finalization flow

Do not save the user’s rough task description directly as an unstructured production prompt.

When the user creates or edits the Quick Task:

1. accept the user’s rough task description;
2. analyze it using a cost-efficient OpenAI model;
3. convert it into a concise, precise, production-ready task instruction;
4. identify the supported context types:

   * text;
   * image;
   * text and image;
5. identify the expected output type:

   * replacement text;
   * analysis;
   * structured data;
   * suggested action;
6. identify whether it may propose external execution;
7. detect instructions that attempt to bypass safety or confirmation policy;
8. show the finalized definition to the user;
9. allow manual editing;
10. require explicit approval;
11. validate the final typed definition;
12. persist it atomically.

```json
{
  "version": 1,
  "name": "Prompt Optimizer",
  "instruction": "Rewrite the selected rough prompt into a precise, structured, implementation-ready prompt while preserving every original requirement and avoiding invented details.",
  "supportedContext": ["text"],
  "outputMode": "replacement_text",

```

Do not allow a Quick Task definition to modify or override:

* action risk classification;
* confirmation requirements;
* API-key handling;

* browser bridge security;
* secret redaction;

### When no selected text exists and the task supports images

Open a compact Quick Task input panel near the cursor.

# 15. One-Shot Screen Analysis Versus Navigation Guidance

These must be different behaviors.

## One-shot analysis

* “What plant is this?”
* “Which animal is this?”
* “What does this error say?”
* “What is on my screen?”
* “Explain this chart.”
* “Which button is this?”
* “Read this image.”

Behavior:

1. capture one approved image;
2. analyze it;
3. answer;
4. discard it from memory when no longer needed;
5. do not start continuous monitoring.

## Navigation Guidance Mode

Use Navigation Guidance Mode when the user asks something like:

* “Help me navigate this website.”
* “Guide me through creating an API key.”
* “Tell me where to click next.”
* “Walk me through this form.”
* “Help me configure this dashboard.”
* “Stay with me while I complete this process.”

Navigation Guidance Mode should:

1. visibly announce that guided screen observation is starting;
2. capture only the active display or active application, according to the user’s setting;
3. analyze the initial state;
4. give one clear next step;
5. wait for the user’s click, command, or confirmation;
6. capture the changed state;
7. verify whether the expected transition occurred;
8. provide the next step;
9. continue until completion, cancellation, inactivity timeout, or an unsafe boundary.

Do not continuously stream full-resolution screenshots at a fixed high rate.

Use an event-driven approach:

* detect user click or navigation event;
* debounce;
* wait for meaningful UI stabilization;
* compare screenshot hashes or browser state;
* avoid sending duplicate frames;
* resize and compress images appropriately;
* send accessibility or DOM summaries when they can replace pixels;
* send a new frame only after meaningful change or explicit request.

The UI must always show when guidance capture is active.

Stopping must be immediate through:

* the orb stop control;
* the Live Mode trigger;
* Escape;
* the global Cancel shortcut;
* voice commands such as “stop watching” or “I’m done.”

Never run navigation monitoring invisibly in the background.

# 19. Take Action Engine

The current Take Action implementation is not reliable enough. Rebuild it as a strict, verifiable action system.

## Critical principle

The model proposes typed actions.

Deterministic local code validates and executes them.

Never execute arbitrary generated JavaScript, PowerShell, shell commands, keyboard macros, selectors, or natural-language instructions.

## Action plan schema

Use a versioned schema similar to:

```json
{
  "version": 1,
  "goal": "Fill the visible application form",
  "contextFingerprint": "bounded identifier",
  "risk": "low|medium|high",
  "requiresConfirmation": true,
  "steps": [
    {
      "id": "step-1",
      "type": "focus_field|set_value|choose_option|toggle_checkbox|click|replace_selection|insert_text|scroll",
      "target": {
        "strategy": "label|role|name|stable_selector|accessibility",
        "value": "Email address"
      },
      "value": "example value",
      "expectedOutcome": "Field contains the value"
    }
  ]
}
```

Validate every plan before execution.

Reject:

* unknown action types;
* unsupported target strategies;
* untrusted scripts;
* hidden or offscreen destructive targets;
* stale context fingerprints;
* cross-origin assumptions;
* actions outside the active tab or approved window;
* plans that exceed bounded step limits.

## Execution priority

For browser workflows:

1. authenticated Chromium extension in the active real tab;
2. DOM and accessibility-based actions;
3. tightly constrained native UI Automation only when necessary;
4. never use unrestricted coordinate clicking as the default.

## Verification

After every meaningful action:

* inspect the DOM, accessibility tree, current value, URL, title, or screenshot;
* verify the expected state;
* stop on mismatch;
* do not claim success based only on sending an input event.

## Undo and recovery

Where practical:

* preserve previous text values;
* expose Undo for text replacement and field filling;
* stop remaining steps after failure;
* explain the exact failed step;
* allow the user to resume after correcting the page.

# 24. Permission and Safety Policy

Do not scatter confirmation decisions across individual tools.

* execute unrelated actions;
* upload files;

* send private data;

* bypass confirmation;

# 26. Code-Quality Limits

Avoid oversized classes and modules.

Guidelines:

* no single “god service” coordinating the entire product;
* no hard-coded 500-line system prompt inside a transport class;
* no 1,000-line automation switch statement;
* no UI code-behind containing API, storage, and action logic;
* no Settings window containing all configuration logic in one file;
* no provider-specific names in domain models;
* no duplicated action schemas between C# and TypeScript;
* no arbitrary JSON dictionaries where typed contracts are possible;
* no catch-all exception swallowing;
* no synchronous network calls on the UI thread;
* no fire-and-forget task without error observation;
* no static mutable global session state;
* no secrets in command-line arguments;
* no fixed localhost ports without collision handling;
* no unauthenticated local HTTP endpoints.

Prefer:

* small cohesive services;
* typed interfaces;
* immutable records where useful;
* cancellation tokens;
* bounded queues;
* state reducers/state machines;
* source-generated JSON serialization where appropriate;
* structured logging;
* dependency injection;
* reusable schema generation;
* explicit timeouts;
* graceful degradation;
* reusable design tokens;
* reusable validated settings controls;
* section-specific settings view models;
* explicit dirty-state and save-state handling.

As a guideline, investigate any source file approaching 400–500 lines and split it when responsibilities differ.

# 27. Local Communication Security

Any localhost or native bridge must:

* bind only to loopback;
